## Successes

1. **Exceptional velocity** — 7/7 stories completed in a single day with 100% acceptance criteria met
2. **Discriminated union tile type system** — enabled clean type narrowing throughout all subsequent stories (draw validation, Joker restriction checks, etc.)
3. **Validate-then-mutate pattern** — held perfectly across all action handlers with zero exceptions for rule violations
4. **Testing culture** — grew from 0 to 162 tests with co-located test files, comprehensive fixtures, and reusable helpers (`createPlayState()`, `getPlayerBySeat()`, tile builders)
5. **Story intelligence chain** — each story's dev notes captured learnings for the next story, compounding knowledge and eliminating rediscovery
6. **Package boundary integrity** — shared/ stayed pure (zero runtime deps, no browser/Node APIs), TypeScript project references caught cross-package violations at compile time
7. **turnPhase discrimination** — eliminated need for separate boolean flags, providing elegant state guards
8. **Manual verification** — visual test harness at `/dev/harness` successfully verified full game loop in browser

## Challenges

1. **Review whack-a-mole (Story 1.4)** — 4 rounds of code review for the same category of issue (null guards for corrupt-state detection). Each round found the fix was needed in another location instead of being applied proactively everywhere.
2. **Story scope bleed (Story 1.4)** — pre-implemented Stories 1.5 and 1.6 alongside 1.4. Gained velocity but broke story traceability and blurred completion tracking.
3. **East's first turn gotcha** — `turnPhase: 'discard'` initial state (correct but unintuitive) tripped up test setup in Stories 1.4, 1.6, and 1.7.
4. **Model inconsistency (Story 1.7)** — switching from Opus to Sonnet introduced debug issues not seen in Opus stories: template parse errors, `window is not defined`, loop dispatch pattern mismatches.
5. **Magic numbers and fragile assertions** — hardcoded `99` for wall remainder and `rack[0]` assertions that could flake on Joker-first scenarios, caught in review.

## Key Insights

1. **Epic 2's story boundaries encode TDD methodology** — Story 2.2 writes red tests, Story 2.3 makes them green. Pre-implementation would destroy the safety net. Story boundaries must be respected.
2. **Pattern matching complexity demands upfront design** — the "fix in review" approach that worked for simple action handlers won't scale to wildcard resolution, Joker substitution across 50+ patterns.
3. **Defensive coding patterns should be applied proactively** — when a review identifies a category of issue, sweep the entire codebase for the same pattern in one round.
4. **Model consistency matters on the critical path** — established patterns carry forward more cleanly when the same model is used throughout.
