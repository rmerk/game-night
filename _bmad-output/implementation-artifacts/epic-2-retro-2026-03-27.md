## Successes

1. **TDD methodology (red-green-refactor)** — Story 2.2 wrote 71 red tests as specification, Story 2.3 made them all green. Caught real issues: backtracking Joker allocation that greedy would have missed.
2. **100% previous retro follow-through** — All 3 action items from Epic 1 retro delivered, with measurable improvements (review rounds down from avg 1.4 to 1.2).
3. **Discriminated union result types** — `ExchangeResult`, `ExposureResult`, `ScoringResult` all follow `{ valid: true, ... } | { valid: false, reason }` pattern. Consistent, reusable, ready for Epic 3A consumers.
4. **Story intelligence chain** — Each story's dev notes explicitly referenced previous story learnings, eliminating rediscovery and compounding knowledge.
5. **Model consistency** — Opus 4.6 throughout all 6 stories. No debug noise from model switching.
6. **Card schema spike validation** — A/B/C color-group abstraction validated upfront with 6 edge-case hands. Zero schema issues during implementation.
7. **Story scope discipline** — No pre-implementation, no scope bleed. Every story delivered only its own scope.
8. **Clean review on Story 2.4** — First story across both epics with zero review findings, first-round approval.

## Challenges

1. **False-confidence tests (Story 2.2)** — HIGH finding: value wildcard range tests asserted hardcoded constants instead of validating actual card data. Would have passed even with broken data. Fixed in review, but concerning it shipped initially.
2. **sp-7/wd-1 card encoding ambiguity (Story 2.3)** — Two structurally identical hands at the same point value. Matcher can't distinguish them because input tiles are literally the same. Not caught in schema spike. Test relaxed to accept either match.
3. **Abstract color matching limitation (Story 2.5)** — `matchesGroupIdentity` can't resolve A/B/C colors to concrete suits for group-level concealed checks. Conservative false-rejection behavior for `sp-1`. Documented in code.
4. **Test helper duplication (Stories 2.2-2.5)** — `buildTilesForHand` (~130 lines) copy-pasted across 3 test files before extraction to shared utility in Story 2.5 review.
5. **Manual card data encoding** — 54 hands encoded by hand in `2026.json` with no tooling for yearly updates. 2025 card is only 7 hands for testing.
6. **Lint warning noise** — 9 type assertion warnings in test helpers accepted but never fixed across the epic.

## Key Insights

1. **Spikes prevent architectural surprises but don't catch all data-level ambiguities** — The schema spike validated the type system perfectly but didn't surface the sp-7/wd-1 tile-level ambiguity.
2. **Test quality matters as much as test quantity** — 213 new tests is impressive, but the Story 2.2 HIGH finding shows that assertion substance must be reviewed, not just coverage numbers.
3. **Shared utilities should be extracted on second use, not third** — The buildTilesForHand duplication across 3 files was avoidable.
4. **Conservative false-rejection is acceptable now but creates tech debt for future year cards** — If more hands with mixed concealed/exposed groups using abstract colors appear in 2027+, the limitation becomes a real problem.
5. **Backtracking > greedy for Joker allocation** — Greedy allocation fails for hands where a pung shares natural tiles with a non-joker-eligible pair (e.g., wd-3). Key architectural discovery.
